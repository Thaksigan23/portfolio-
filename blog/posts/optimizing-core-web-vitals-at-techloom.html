Optimizing Core Web Vitals at Techloom.ai

2026-06-18 · Engineering

Working as a Trainee Software Engineer at Techloom.ai (after starting as Development & Marketing Executive) means wearing two hats: shipping clean frontend code and making sure the platform itself performs like the product we sell. When your agency promises high-performance digital experiences, your own site has to lead by example. Our target: keep Largest Contentful Paint (LCP) under 2.0 seconds on average.

Starting with the layout

Performance optimization isn't a final polish step — it starts in the layout. I rebuilt page sections using responsive, reusable UI components so we weren't duplicating heavy markup across routes. Fewer one-off layouts means smaller bundles, more predictable rendering, and easier lazy-loading decisions.

Component-level wins

  • Hero sections: Prioritized the LCP element — usually the headline or hero image — with explicit dimensions to prevent layout shift
  • Images: Used modern formats and size-appropriate sources so mobile doesn't download desktop assets
  • Fonts: Limited weight variants and used font-display: swap to avoid invisible text during load
  • Below-the-fold content: Deferred non-critical sections so the first paint stays fast

Headless CMS integration

Techloom.ai uses a Headless CMS for scalable content delivery. The challenge is that CMS-driven pages can easily become slow if every route fetches too much data at build time or blocks rendering on slow API calls. I helped structure flexible schemas so pages fetch only the fields they need, and server-side rendering handles the initial HTML before client hydration kicks in.

Design-to-code workflow

A big part of my role is translating high-fidelity UI/UX wireframes into production interfaces without losing visual hierarchy. That means matching spacing, typography, and component states precisely — but also pushing back when a design choice hurts performance. A heavy animation on the hero might look great in Figma and still cost 400ms of LCP.

Measuring what matters

I tracked Core Web Vitals in real conditions — not just localhost. Mobile throttling, slow 4G, and cold cache loads reveal problems that dev machines hide. The goal isn't a perfect Lighthouse score; it's consistent sub-2.0s LCP for real users landing on the homepage from social or search.

What I'd do differently

  • Set performance budgets before building new page templates, not after launch
  • Automate image optimization in CI so contributors can't accidentally ship 2MB hero photos
  • Document component performance guidelines so the whole team builds with LCP in mind
